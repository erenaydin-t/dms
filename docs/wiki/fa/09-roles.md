# نقش‌ها و سطوح دسترسی

ماژول DMS از **سه نقش اختصاصی** به‌علاوه نقش استاندارد *Employee* برای خوانندگان استفاده می‌کند. نام نقش‌ها باید همان نام‌های **انگلیسی استاندارد** بمانند — رکورد نقش را تغییر نام ندهید (بخش «ملاحظات ویژه»).

## خلاصه نقش‌ها

| نقش | الزامی؟ | هدف |
|---|---|---|
| **QA Manager** | ✅ بله — نقش کاری | همه تهیه‌کنندگان، بررسی‌کنندگان و تأییدکنندگان QA |
| **DMS Manager** | ✅ بله (دست‌کم یک نفر) | مالک/راهبر ماژول: کنترل کامل شامل حذف، نظارت فرا-واحدی، توزیع فایل مبدأ |
| **System Manager** | داخلی فراپه | راهبری IT؛ همان دسترسی نامحدود DMS Manager |
| **Employee** | اختیاری | خوانندگان: فقط مستندات *تأییدشده و فعال* واحد خودشان |

کاربر `Administrator` همه گیت‌های نقشی و فردی را به‌طور ضمنی رد می‌کند (مسیر اضطراری) — برای استفاده روزمره مناسب نیست، چون رد ممیزی باید افراد واقعی را نام ببرد.

---

## QA Manager

**هدف:** نقش کاری روزمره؛ هر تهیه‌کننده، بررسی‌کننده و تأییدکننده QA باید آن را داشته باشد.

**مجوزهای DocType:** خواندن، نوشتن، ایجاد، ثبت، لغو، اصلاح (Amend)، گزارش، خروجی، چاپ، ایمیل، اشتراک — **بدون حذف**.

**دامنه دید:** نامحدود — همه مستندات در همه واحدها و وضعیت‌ها.

**اقدامات گردش‌کار** (همگی علاوه بر نقش، به «شرط فردی» نیاز دارند — داشتن نقش لازم است اما کافی نیست):

| اقدام | شرط اضافه |
|---|---|
| Submit for Review | باید **تهیه‌کننده** سند باشد (`prepared_by`) |
| Approve as Reviewer / Request Revision (Reviewer) | باید **بررسی‌کننده** تعیین‌شده باشد |
| Approve as QA / Request Revision (QA) | باید **تأییدکننده QA** تعیین‌شده باشد |
| Cancel Revision | باید تهیه‌کننده **یا** تأییدکننده QA باشد؛ فقط روی پیش‌نویس‌های «Create Revision» |

**سایر قابلیت‌ها:** دکمه Create Revision؛ گزینه *Edit Effective Date* (زمان‌بندی/عقب‌بردن تاریخ)؛ بازتولید PDF پایه مفقود هنگام دانلود؛ ویرایش سند در وضعیت‌های **Draft** و **Revision Requested**.

**وابستگی‌ها:** برای اقدام در گردش‌کار، کاربر باید روی همان سند **تعیین شده** باشد؛ بررسی‌کننده و تأییدکننده QA باید **تصویر امضا** روی Employee متصل داشته باشند — وگرنه ذخیره سند مسدود می‌شود.

---

## DMS Manager

**هدف:** مالک ماژول؛ همه اختیارات QA Manager به‌علاوه اختیارات راهبری.

**مجوزهای DocType:** همه موارد QA Manager **به‌علاوه حذف**.

**نقش در وضعیت‌ها:** مستندات در وضعیت‌های *Under Review*، *Pending QA Approval*، *Approved* و *Revision Cancelled* فقط برای DMS Manager قابل ویرایش‌اند (`allow_edit`) — یعنی QA Manager عادی نمی‌تواند سندی را که از دستش خارج شده تغییر دهد.

**قابلیت‌های انحصاری:** *Download Word (clean)* — فایل مبدأ بدون واترمارک (توزیع نسخه ویرایش‌پذیر امتیاز مدیریتی است)؛ حذف پیش‌نویس‌ها (هرگز بازنگری‌های لغوشده — گارد نگهداری برای همه فعال است).

**نکته:** DMS Manager در گذارهای گردش‌کار حضور ندارد — اقدامات گردش‌کار همچنان QA Manager + تعیین‌شدن روی سند می‌خواهند. مالکی که خودش سند می‌نویسد باید **هر دو نقش** را داشته باشد.

---

## System Manager

از دید ماژول نامحدود است (مانند DMS Manager) و گیت‌های تاریخ اجرا و دانلود Word را نیز رد می‌کند. فقط برای راهبری IT اختصاص دهید، نه به‌جای نقش‌های ماژول.

---

## Employee (اعضای واحد)

**هدف:** مصرف‌کنندگان مستندات تحت کنترل.

**مجوزهای DocType:** فقط خواندن، گزارش و چاپ.

**دامنه دید (با فیلتر SQL + بررسی تک‌سند):** فقط مستندات **تأییدشده (ثبت‌شده)، فعال و متعلق به واحد(های) خود کاربر** — به‌علاوه هر سندی که *نام خود او* را به‌عنوان تهیه‌کننده/بررسی‌کننده/تأییدکننده دارد (در هر وضعیتی). پیش‌نویس‌ها، مستندات در انتظار اجرا، منسوخ‌ها و بازنگری‌های لغوشده دیگران هرگز دیده نمی‌شوند.

**دانلود:** PDF کنترل‌شده/کنترل‌نشده/ساده برای مستندات قابل مشاهده. هرگز فایل Word تمیز؛ بازتولید PDF پایه هم در دسترسشان نیست.

## تعامل نقش‌ها در طول چرخه عمر

```
 تهیه‌کننده (QA Manager)      بررسی‌کننده (QA Manager)      تأییدکننده QA (QA Manager)
        │ ایجاد + پیوست             │                             │
        │ Submit for Review ───────▶│ Approve as Reviewer ───────▶│ Approve as QA
        │ ◀── Request Revision ─────┘ ◀── Request Revision ───────┘        │
        │                                                        ثبت خودکار، رندر،
        │                                                        امضا، مهر، فعال‌سازی
        ▼                                                                 ▼
   (ویرایش فقط در Draft /                                       کارکنان واحد:
    Revision Requested)                                         خواندن و چاپ
                                                                نسخه کنترل‌شده
   DMS Manager: مداخله در هر وضعیت، حذف پیش‌نویس،
   توزیع فایل Word تمیز، اختیارات راهبری.
```

همین الگوی سه‌نفره برای هر بازنگری تکرار می‌شود؛ نسخه قبلی تا لحظه تأیید بازنگری (یا فعال‌سازی، اگر تاریخ آینده دارد) برای واحدش قابل خواندن می‌ماند و پس از آن، اعضا خودکار فقط نسخه جدید را می‌بینند.

## ملاحظات ویژه

- **رکورد نقش‌ها را هرگز تغییر نام ندهید.** تغییر نام، گذارهای گردش‌کار را با خود می‌برد در حالی که مجوزهای DocType و بررسی‌های کد، نام استاندارد را می‌خواهند — نتیجه، قفل‌شدن همه کاربران است («Not a valid Workflow Action»). از نسخه 2.3.1، اجرای `bench migrate` گردش‌کار را به نقش‌های استاندارد بازمی‌گرداند. برای نمایش فارسی، از **Translation** (منوی Setup ← Translation) استفاده کنید.
- **چند نقش برای یک نفر** بلامانع است (مثلاً بررسی‌کننده = تأییدکننده QA)، اما هر نقش باید روی سند تعیین شده باشد و شرط‌های فردی اجازه تأیید مرحله‌ای را که فرد مسئولش نیست نمی‌دهند.
- **Self-approval** در سطح فراپه عمداً مجاز است، چون تهیه‌کننده مالک پیش‌نویسی است که ثبت می‌کند؛ تفکیک وظایف با *تعیین افراد* روی سند اعمال می‌شود، نه مالکیت رکورد.
- **عضویت واحد** برای دید Employee از فیلد department رکورد Employee او خوانده می‌شود؛ کاربر بدون Employee (یا بدون واحد) فقط مستندات نام‌برنده خودش را می‌بیند.
- گردش‌کار `GMP Document Workflow` باید **Active** بماند؛ غیرفعال‌شدن آن گذارها را برای همه نقش‌ها می‌شکند.
