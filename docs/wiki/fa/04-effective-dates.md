# تاریخ اجرا و زمان‌بندی

فیلد `effective_date` (تاریخ اجرا) از الگوی آشنای **Posting Date** در ERPNext پیروی می‌کند.

## حالت پیش‌فرض: کنترل توسط سیستم

اگر گزینه **Edit Effective Date** خاموش باشد (پیش‌فرض)، سیستم هنگام تأیید QA **تاریخ تأیید** را درج می‌کند. هر مقدار دستی هنگام ذخیره به‌صورت خاموش حذف می‌شود — دقیقاً مانند رفتار ERPNext با Posting Date وقتی «Edit Posting Date and Time» خاموش است. تاریخ‌های مشتق‌شده خودکارند:

- `expiry_date` (انقضا) = تاریخ اجرا + دوره اعتبار (۲/۳/۵ سال)
- `next_revision_date` (بازنگری بعدی) = انقضا − ۱ ماه

## حالت دستی: تاریخ گذشته یا آینده

گزینه **Edit Effective Date** را روشن کنید (فقط برای QA Manager / DMS Manager / System Manager — در سمت سرور اجبار می‌شود) و پیش از تأیید، تاریخ دلخواه را وارد کنید:

- **تاریخ گذشته:** سند بلافاصله پس از تأیید معتبر می‌شود و انقضا/بازنگری بعدی از همان تاریخ گذشته محاسبه می‌گردند.
- **تاریخ آینده:** سند تأیید می‌شود اما **در انتظار اجرا** می‌ماند:

| در حالت انتظار | |
|---|---|
| `docstatus` | 1 (ثبت‌شده) |
| `workflow_status` | Approved |
| `is_active` | **0** — هنوز نسخه معتبر نیست |
| واترمارک همه نسخه‌های PDF | **NOT YET EFFECTIVE** |
| نسخه قبلی (جریان بازنگری) | **همچنان تأییدشده، فعال و معتبر** |
| نشانگر فهرست | نارنجی *«Effective DD-MM-YYYY»* |

اگر حالت دستی روشن باشد ولی تاریخی وارد نشده باشد، تأیید QA با پیام روشن مسدود می‌شود.

## فعال‌سازی خودکار

زمان‌بند روزانه `activate_effective_documents` (که **پیش از** زمان‌بند انقضا اجرا می‌شود) هر سند در انتظاری را که تاریخش فرا رسیده فعال می‌کند:

1. `is_active` ← 1 — سند نسخه معتبر می‌شود
2. نسخه قبلی جایگزین‌شده به **منسوخ** می‌رود
3. ارجاع‌های وابسته منتقل می‌شوند
4. یادداشت ممیزی روی هر دو رکورد ثبت می‌شود

بین نیمه‌شب و اجرای زمان‌بند، سندِ موعدرسیده همچنان «در انتظار اجرا» خوانده می‌شود — هرگز به‌اشتباه «منسوخ» یا «کنترل‌شده» نمایش داده نمی‌شود. فعال‌سازی فقط بر عهده زمان‌بند است تا جابه‌جایی، یکپارچه و قابل ممیزی بماند.

## انقضا

زمان‌بند روزانه `expire_gmp_documents` هر سند فعالِ گذشته از `expiry_date` را منسوخ می‌کند (docstatus = 1 می‌ماند و دانلودها واترمارک OBSOLETE می‌گیرند). برای جلوگیری از خلأ پوشش، بازنگری جایگزین را پیش از `next_revision_date` آغاز کنید.
